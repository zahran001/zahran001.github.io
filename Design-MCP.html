Spec to Component

A RAG pipeline that grounds AI-generated UI in your real design system. Validated end-to-end on Chakra UI v3.

Description

Ask an AI coding agent for a UI component and it often invents something that looks plausible but quietly breaks your design system: wrong props, deprecated APIs, inconsistent composition. The reason is structural. Most companies run their own internal design system, not a generic one, so whatever the model absorbed in training is the wrong thing to trust. I saw this concretely when an agent generated Chakra that wouldn't compile, and a separate evaluation pass rated the broken output as correct: the model wasn't careless, it was faithful to the wrong version.

Spec to Component grounds agent-generated UI in the real system of record. It crawls a design system's own docs, chunks and embeds them, retrieves the relevant pieces at generation time, and gates the output on signals a compiler can verify rather than an LLM's opinion. Point the crawler at a different design system and the same retrieval and validation loop applies.

Chakra UI v3 is the validated proof case, chosen because its v2→v3 breaking change is exactly the kind of drift that trips up a model's memory: colorScheme became colorPalette, and monolithic components became slot-based composition. 897 chunks, 50 components, generation gated on tsc and lint, 100% valid single-shot.

Features

  1. Built a design-system-agnostic RAG pipeline (Playwright BFS crawler → normalizer → embedder → retriever) that turns any design system's own docs into typed, retrievable chunks, then pointed it at Chakra UI v3 as the validated proof case. The result: 897 typed chunks (code examples, prop references, overviews, capabilities) across 50 components, embedded with OpenAI text-embedding-3-small into Qdrant for reserved-slot retrieval.
  2. Found that an LLM-as-judge inverts on Chakra v3, preferring the familiar v2 API and marking correct v3 as "outdated." So I replaced it with three objective, compiler-verifiable gates (a tsc type-check, a v2-smell lint, a composition lint) plus a bounded self-heal loop that feeds tsc diagnostics and smell-named hints back to the model.
  3. Ran a grounded-vs-no-context A/B and a six-pass correction-loop experiment that took grounded tsc-pass from 47% to 100% single-shot on DeepSeek V4 (swapped in from gpt-4o because it's cheaper and, with grounding, more reliable). Recorded honest negatives too: a naïve tsc-feedback repair loop failed completely, and a thinking-mode A/B showed no objective gain at +46% latency.
  4. Deployed a single-origin SPA + API to Google Cloud Run (Docker, slimmed 539→462 MB) with Qdrant Cloud for retrieval and a client-side Sandpack live preview, keeping the deploy image Chromium-free.

Architecture

Offline — build the corpus:
Crawl (Playwright) → Normalize (typed chunks) → Embed (OpenAI, 1536-d) → Qdrant (897 vectors)

Per request:
NL request → Retrieve (reserved-slot) → Generate (DeepSeek V4, grounded) → tsc self-heal (bounded loop) → Objective gates (tsc · smell · composition) → Live preview (Sandpack)

One container serves the SPA + API (single origin, no CORS). Generation runs on DeepSeek V4; embeddings stay on OpenAI (the query vector must match the corpus). Render-check is off in prod: Sandpack renders client-side and tsc stays the gate.

Engineering Reflection

The single most useful finding in this project was a negative one: I built an LLM-as-judge to score the generated components and found it inverts on Chakra v3 — it rates correct v3 composition as "outdated" and incorrect v2 as "current," because its training data is v2-dominant. You can't evaluate API-drift compliance with a model that shares the drift bias, which is why tsc + a v2-smell lint + a composition lint became the trusted spine and the judge got demoted to a secondary signal.

The correction loop had its own honest negative: my first instinct for fixing type errors was to feed raw tsc compiler output back to the model for repair. It failed completely, 0 of 7 cases fixed across 14 attempts, because TypeScript's JSX-prop errors don't name the bad prop. What actually worked was reusing the v2→v3 rename map to turn those coarse errors into named-prop hints ("this looks like the old colorScheme prop; v3 uses colorPalette"), which took grounded tsc-pass from 53% to 73% — the single biggest lift in the whole experiment.

DeepSeek's "thinking mode" was the obvious next lever to pull, so I A/B'd it instead of assuming. No objective-gate gain anywhere (everything was already at ceiling), and +46% latency for nothing. Shipped off, with the numbers to back that call rather than a guess.

I also swapped generation from gpt-4o to the cheaper DeepSeek V4, and with grounding in place, it beat the gpt-4o baseline on the objective gates (93%→100% single-shot). The reliability came from retrieval + validation, not a bigger or pricier model, which is the whole thesis of the project restated as a result.
